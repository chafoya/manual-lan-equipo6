📡 ¿Por qué Topología Estrella?

La topología estrella fue elegida porque si un cable falla, solo ese equipo pierde conexión sin afectar al resto. Además facilita agregar nuevos equipos sin reconfigurar toda la red y el diagnóstico de problemas es más sencillo.

El servidor es el equipo central de la red. Tiene IP fija permanente porque todos los clientes apuntan a esta dirección para acceder a carpetas compartidas y recibir administración remota.

🔧 Configuración de IP Estática — Paso a Paso
1
Abrir configuración de red
Win + R → escribir ncpa.cpl → Enter. Se abre la ventana de adaptadores de red.
2
Propiedades del adaptador Ethernet
Click derecho en Ethernet → Propiedades → doble click en TCP/IPv4.
3
Ingresar los datos exactos
Seleccionar "Usar la siguiente dirección IP" y llenar:

4
Cambiar nombre del equipo a SERVER-LAN01
Click derecho Inicio → Sistema → Cambiar nombre de este equipo → escribir SERVER-LAN01 → Reiniciar.

Verificar con ipconfig

1

2
Instalar como Tutor en el servidor

Abrir Consola de Tutor

💻 NetSupport School — Instalación Student (Clientes)
1
Acceder al instalador desde la red
Abrir Explorador → escribir \\192.168.10.10\RECURSOS → abrir setup.exe como administrador.
2
Seleccionar componente STUDENT únicamente
En selección de componentes marcar SOLO Student → NO instalar Tutor ni Tech. Esto instala el servicio que permite la conexión desde el servidor.
3
Verificar conexión automática
Al terminar la instalación, el cliente aparece automáticamente en la Consola de Tutor del servidor. No requiere configuración adicional de llaves ni contraseñas.

Tareas Programadas y Scripts
📄 Script: backup.bat

Ubicación: C:\Scripts\backup.bat

:: Backup Automático — SERVER-LAN01
@echo off
set FECHA=%date:~6,4%-%date:~3,2%-%date:~0,2%
set ORIGEN=C:\Compartido
set DESTINO=C:\Compartido\BACKUP\backup_%FECHA%
echo Iniciando backup: %FECHA%
xcopy "%ORIGEN%\PUBLICO" "%DESTINO%\PUBLICO" /E /I /Y
xcopy "%ORIGEN%\PROYECTOS" "%DESTINO%\PROYECTOS" /E /I /Y
echo Backup completado exitosamente.
pause
✅
Script probado en producción. Copió exitosamente 2 archivos de PUBLICO: index.html y manual_servidor_administrador.html

SECCIÓN 07 //
Problemas Encontrados y Soluciones
📚
Nota pedagógica: Los errores encontrados durante el proyecto fueron oportunidades de aprendizaje real. Cada problema diagnosticado y resuelto refuerza las competencias de soporte técnico en redes.
❌ PROBLEMA 1: Botón "Aplicar" en gris al compartir carpetas
Causa: El servicio "Servidor" de Windows estaba detenido y la detección de redes estaba desactivada.
Solución: Activar servicio "Servidor" en services.msc → Tipo de inicio: Automático → Iniciar. Luego en Panel de control → Centro de redes → activar detección de redes y uso compartido de archivos.
❌ PROBLEMA 2: Veyon — Error de autenticación en PC-CLIENTE1
Causa: La llave pública admin_public.pem no fue importada correctamente. El Veyon Configurator no se ejecutó como administrador.
Solución intentada: Desinstalar y reinstalar Veyon Service, importar llave vía CMD con veyon-cli authkeys import, copiar .pem manualmente a C:\ProgramData\Veyon\keys\public\admin\
Resultado final: Se migró a NetSupport School que no requiere intercambio de llaves. Veyon quedó funcional solo en 2 de 4 clientes.
❌ PROBLEMA 3: PC con Ethernet en "medios desconectados"
Causa: El cable UTP no estaba conectado al switch antes de configurar la IP estática.
Solución: Conectar físicamente el cable al switch → verificar LED verde en el puerto → luego configurar IP. El adaptador pasó de "desconectado" a activo.
❌ PROBLEMA 4: Los equipos no aparecían en "Red" del Explorador
Causa: El servicio "Function Discovery Resource Publication" estaba deshabilitado en algunos equipos.
Solución: Activar en services.msc. En Windows 11 algunos equipos no tenían este servicio disponible; se resolvió activando la detección de redes manualmente en la configuración de red avanzada.
❌ PROBLEMA 5: Veyon CLI [FAIL] — "No se pudo leer el archivo de entrada"
Causa: La ruta UNC \\192.168.10.10\RECURSOS\admin_public.pem no era accesible desde el CMD del cliente en ese momento.
Solución aplicada: Copiar el archivo .pem físicamente a la PC cliente y ejecutar el comando con ruta local. Como alternativa definitiva se adoptó NetSupport School.
🔍 Proceso de Diagnóstico de Conectividad (Q-011)
1
Verificar el cable físico
Confirmar LED verde en el puerto del switch. Sin LED = sin señal física.
2
Ping al loopback
ping 127.0.0.1 — si falla, problema en el adaptador de red del equipo.
3
Verificar IP con ipconfig
Si aparece 169.x.x.x → IP no configurada. Debe ser 192.168.10.XX con máscara 255.255.255.0.
4
Ping al servidor
ping 192.168.10.10 — si falla: verificar Firewall o cable.
5
Revisar Firewall
Habilitar reglas: ICMPv4, SMB y detección de redes en las reglas de entrada del Firewall de Windows.

SECCIÓN 08 //
Sorpresa del Equipo: Juego en Red LAN
⚔ PIXEL WARS — Trivia Multijugador LAN

Juego de trivia multijugador desarrollado específicamente para este proyecto. Funciona 100% en red local sin internet. El servidor corre un backend en Python y todos los clientes se conectan desde su navegador escribiendo la IP del servidor.

CARACTERÍSTICAS TÉCNICAS
  • 🐍 Backend: Python 3 (HTTP server nativo)
  • 🌐 Frontend: HTML/CSS/JS puro
  • 📡 Comunicación: polling cada 800ms
  • 👥 Hasta 8 jugadores simultáneos
  • ❓ 8 preguntas sobre Redes LAN
  • ⏱️ 20 segundos por pregunta
  • 🏆 Puntos + bonus por velocidad
CÓMO JUGAR EN RED LAN
:: En el servidor — doble click en:
INICIAR_JUEGO.bat

:: Clientes abren su navegador y van a:
http://192.168.10.10:8000
Solo el servidor (SERVER-LAN01) puede iniciar la partida. Los clientes se unen con su nombre y esperan en el lobby.
⛏️ Alternativa: Minecraft Multijugador en LAN

Minecraft en modo LAN valida que la red funciona para comunicación en tiempo real. El juego descubre servidores automáticamente en la misma subred 192.168.10.0/24.

1
Distribuir instalador de Minecraft desde \\192.168.10.10\RECURSOS a todos los equipos.
2
Instalar la misma versión en todos los equipos (ej: 1.20.x) para garantizar compatibilidad.
3
En el servidor: crear mundo → Escape → Abrir a LAN → configurar modo de juego → Iniciar mundo en LAN.
4
En los clientes: abrir Minecraft → Multijugador → el servidor aparece automáticamente en la lista.
5
Conectarse y jugar juntos. La red LAN queda validada si todos pueden jugar simultáneamente sin lag.

SECCIÓN 09 //
Áreas de Mejora
01
Completar implementación de Veyon: Investigar la configuración del backend de autenticación para lograr conexión exitosa en los 4 clientes. Veyon ofrece más control que NetSupport School para entornos educativos.
02
Implementar servidor DHCP: En lugar de configurar IPs estáticas manualmente, instalar un servidor DHCP que las asigne automáticamente. Reduciría errores humanos y facilitaría agregar nuevos equipos.
03
Servidor de archivos dedicado: Usar un NAS o Windows Server para centralizar mejor los recursos compartidos en lugar del equipo del helper.
04
Firewall centralizado: Usar Group Policy o scripts PowerShell para aplicar reglas de Firewall automáticamente en todos los equipos sin configuración manual individual.
05
Permisos diferenciados: Configurar usuarios y contraseñas individuales para las carpetas compartidas, en lugar de acceso abierto a todos los equipos de la red.
06
Monitoreo de red: Implementar herramientas como Wireshark o PRTG para analizar el tráfico, detectar cuellos de botella y diagnosticar problemas de conectividad de manera profesional.

Conclusiones y Logros

La implementación del proyecto de Red LAN fue una experiencia altamente enriquecedora que permitió al equipo aplicar de forma práctica los conceptos teóricos aprendidos durante el ciclo académico.
